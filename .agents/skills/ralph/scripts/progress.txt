## Codebase Patterns
- Cat definitions live in src/game/cats/definitions/<CatType>.ts; all exported via CAT_REGISTRY Map<CatType,CatDefinition> in definitions/index.ts — systems should import CAT_REGISTRY, not individual files
- CatDefinition.behavior.params is Record<string, number|string|boolean> for cat-specific config (no per-cat union types needed in the interface)
- OxygenState is added/removed by WaterSystem alongside SwimmingState — both live and die together on water entry/exit
- OxygenSystem queries OxygenState+SwimmingState+PlayerControlled; OxygenState is always paired with SwimmingState (no need for null guards when both appear in query)
- SceneManager.addMesh() now supports opacity field; setMeshColor/setMeshOpacity allow runtime material mutation without recreating meshes
- CatPlacementSystem runs in the variable-rate render pass (not fixed-step loop) so clicks are processed exactly once per render frame
- GameState is instantiated in Game.ts before CatCompanionManager (dependency order); CatCompanionManager gets a `() => playerEntity` getter to avoid circular refs
- UIManager.setCatCatalog() must be called after map load (catalog comes from CatCompanionManager.getCatalog() which reads CAT_REGISTRY)
- HUD.setCatalog() builds cat slot DOM elements lazily — call it once at init, then update via setCatBar() each frame
- HUD.update(dt, oxygenPercent, health, maxHealth) — UIManager.update(dt, HUDState?) is the call site; Game.buildHUDState() assembles the snapshot each render frame
- Player health lives in PlayerControlled.health/maxHealth (integer hp); OxygenState.healthDrainAccum accumulates sub-frame losses to avoid float rounding
- CSS keyframe for oxygen warning pulse is injected once via <style id="__oxygen-pulse-style"> to avoid duplicates
- Water triggers use collisionLayer=1/mask=1 (player layer) so CollisionSystem fires trigger events; floor tiles use layer=2/mask=0 (PhysicsEngine raycasts only)
- SwimmingState lifecycle is event-driven (trigger:enter/exit from CollisionSystem); MovementSystem applies swim physics by querying for SwimmingState each frame — no state machine
- PhysicsEngine.setGravityEnabled(handle, bool) controls per-body gravity; must be called when adding/removing SwimmingState
- Terrain entities are created before the player, so terrain entity IDs are always smaller; in pairKey(a,b) the smaller id comes first → water entity is usually "trigger" in events
- Game loop order: movementSystem → physics.step() → collisionSystem → waterSystem (WaterSystem reacts to trigger events emitted by CollisionSystem)
- Config changes: swimSpeed/swimSpeedVertical replaced with swimSpeedSurface(3.2), swimSpeedDive(2.0), swimSpeedAscend(2.5), swimSpeedVertical(1.5), swimAcceleration(0.5)
